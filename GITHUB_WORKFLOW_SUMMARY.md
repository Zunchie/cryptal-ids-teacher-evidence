# GitHub Workflow Summary

GitHub was used as the project’s main evidence and workflow system.

## Issues

Issues were used to define bounded tasks, diagnostics, repair work, decision points, and validation targets. Issue titles show the project moving through contracts, route repairs, authority checks, controlled research steps, and result reviews.

## Pull Requests

Pull requests were used as small reviewable units. PR bodies often stated purpose, scope, validation, forbidden areas, and reviewer summaries. This made process visible and created a record of what was attempted.

## Checks And Validation

PR evidence references checks such as review-packet building, project gates, smoke checks, repo sanity checks, and control-file verification. Local validation summaries in PR bodies also reference unit tests, format checks, and structured-data checks.

## Human-Gated Merges

The workflow repeatedly emphasized human review before merge. This matters because the project was about controlled automation, not unsupervised action.

## Failure To Repair

Failures became repair tasks. Examples include unsupported next actions, missing next-action markers, route-selection mistakes, semantic failures, and authority graph problems. This is strong evidence of learning because it shows iteration, not just final output.
